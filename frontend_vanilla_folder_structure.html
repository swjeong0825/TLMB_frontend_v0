Overview

The frontend is a static browser app. It renders the TLM pages, calls the chat-to-intent server for natural-language chat, and submits confirmed write forms directly to backend_main.

Static contract

No build step

Every file is served as written. Do not add a framework, bundler, transpiler, ES module graph, or committed runtime dependency.

Page contract

One boot script

Each page has one entry/controller script under js/. Complex pages can load support modules first, but those modules do not boot the page.

Localization

EN and KO dictionaries

Runtime UI copy belongs in js/i18n/en.js and js/i18n/ko.js. js/i18n.js stays the registry/helper module.

Pages

Page folders are the user-facing routes. Page entry scripts live under js/; support modules live under js/<page>/ when a page needs more structure.

Route HTML file Entry script Primary responsibility
/ index.html Inline shell init only Landing page and navigation to create/find/demo flows.
/create-league/ create-league/index.html js/create-league.js Create a league, collect optional roster, render host/player URLs.
/find-league/ find-league/index.html js/find-league.js Search leagues by title prefix and open a selected player chat.
/find-league-prefix/?prefix=... find-league-prefix/index.html js/find-league-prefix.js Auto-run title-prefix search from the URL query string.
/league/?league_id=... league/index.html js/chat.js Per-league chat. Player mode by default, admin mode with host_token.
/demo demo/index.html js/chat.js Demo chat page with preconfigured sample league values.
Page rule: support modules can prepare helpers and attach them to a namespace, but the page entry script owns booting, route parsing, state creation, and event wiring.

CSS Structure

Every page links only /css/styles.css. That file is an ordered manifest. Chat and create-league have their own nested manifests so large surfaces stay split without changing the HTML links.

Global manifest

css/styles.css

Controls the top-level cascade. Keep this file as import ordering only; place actual rules in focused modules.

JavaScript Structure

JavaScript is organized by shared support scripts, page entry scripts, and optional page-specific support modules. Files are loaded directly by HTML, so dependency order is explicit in each page.

Shared runtime

js/ shared files

config.js owns production API defaults and query overrides.

user-facing-errors.js maps technical errors to localized user copy.

en.js and ko.js register dictionaries with window.TLCHAT_I18N. Runtime UI strings should be added to both files.

Where to Edit

Start in the narrowest owner for the behavior you are changing. The entry scripts should stay focused on booting, state, route parsing, and orchestration.

Frontend Rules

These are the boundaries that keep the static frontend maintainable.

  • No build step. No framework, bundler, transpiler, ES module import/export, or committed runtime dependencies.
  • One entry script per page. Support modules are ordered no-boot IIFEs attached to window.TLCHAT_* namespaces.
  • Runtime UI strings must be localized. Add English and Korean entries under js/i18n/, then render through TLCHAT_I18N.
  • The chat page never writes through the chat server. Write intents render confirmation forms; confirmed requests go directly to backend_main.
  • Admin writes use /admin/* URLs. route.hostToken is the role signal. Player writes use non-admin /leagues/... URLs.
  • Do not show technical IDs to users. IDs stay in URLs, request bodies, and data-* attributes, not visible success/error UI.
Verification habit: for JavaScript changes, run node --check on touched files. For page behavior, serve frontend/frontend_vanilla locally and smoke the affected route.
